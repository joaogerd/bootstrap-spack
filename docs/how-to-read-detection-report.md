# How to Read the Detection Report

## Purpose

`detection-report.txt` is the main diagnostic artifact generated by `bootstrap-spack`.

If something looks wrong in the final YAML files, this report should be the first thing you inspect.

It is the place where the tool explains:

- what it detected
- what it validated
- what it inferred
- what policy it derived
- which overrides were applied
- why the final output looks the way it does

---

## Where to find it

After running the tool, the report is written to:

```text
<output-dir>/detection-report.txt
```

Example:

```bash
bootstrap --config env/egeon.yaml --output-dir out --debug
```

Then inspect:

```text
out/detection-report.txt
```

---

## Recommended reading order

Read the report in this order.

1. detection status
2. toolchain summary
3. generated specs
4. detected facts
5. derived policy
6. policy authority
7. policy trace
8. warnings

This order goes from raw observation to final rendered meaning.

---

## 1. Detection section

This section tells you whether requested packages were found and validated.

Typical questions to ask:

- Was the package found?
- Was validation successful?
- Which prefix was detected?
- Does the validation result match the intended machine environment?

Examples of useful interpretation:

- `OK openmpi ...` means the MPI wrapper and metadata were coherent enough for the validator
- `OK hdf5 ...` means HDF5 tools and metadata were detected consistently
- failures here usually mean missing tools, wrong modules, or mismatched wrappers

If detection is wrong here, anything derived later will also be wrong.

---

## 2. Toolchain section

This section summarizes the consistency of the detected stack.

Typical questions to ask:

- Do the compiler, MPI and libraries look coherent together?
- Is the environment mixing incompatible components?

If the toolchain summary shows inconsistency, do not trust the rendered artifacts yet.
Fix the environment first.

---

## 3. Specs section

This section shows the reconstructed external package specs.

Typical questions to ask:

- Are versions correct?
- Are prefixes correct?
- Were dependencies inferred reasonably?
- Are assumptions acceptable?

Examples:

- `^openmpi@4.1.1` inferred from dynamic linkage may be correct and useful
- a spec with many assumptions should be reviewed more carefully than one with direct evidence

This section is especially important for:

- checking whether HDF5 / NetCDF dependencies were reconstructed correctly
- checking whether MPI linkage was understood correctly

---

## 4. Facts section

This section describes what the tool believes the host actually exposed.

Typical content includes:

- loaded modules
- detected compiler
- detected runtime config
- explicit platform facts

This is where you verify whether the tool understood the real machine correctly.

Examples of good questions:

- Did it detect the correct compiler family and version?
- Did it detect the expected normalized OS?
- Did it detect the expected target architecture?
- Did it capture the expected module environment?

A key 0.4.1 rule is:

- facts are about the host
- facts are not automatically the same thing as final policy

---

## 5. Policy section

This section tells you what final policy the tool derived from configuration plus detected facts.

Typical things to inspect:

- selected providers
- final runtime policy
- common module policy
- template policy
- final platform policy values

This is where you answer:

- what does the tool intend to render into final YAML files?

The most important conceptual rule is:

- facts explain the machine
- policy explains the generated site

---

## 6. Policy authority section

This section explains which authority produced an important policy value.

Typical metadata includes:

- `key`
- `value`
- `source`
- `rationale`
- `confidence`
- `precedence_rank`
- `overridden_by`
- `supersedes_source`
- `fallback_used`

This section is critical when you need to understand why a value won.

Examples:

- `source = detection` means the value came from observed machine facts
- `source = override` means configuration deliberately replaced a detected or derived value
- `supersedes_source = detection` means an override explicitly replaced what the machine reported

This is the best section for auditing whether an override was intentional and traceable.

---

## 7. Policy trace section

This section explains the decision path.

It is useful when you need to reconstruct the reasoning path that led to the final policy.

Typical things you can learn here:

- which policy values were derived from configuration
- which came from detection
- which were overridden
- which fallbacks were used

If the policy looks surprising, the trace usually tells you where the surprise entered.

---

## 8. Warnings

Warnings should never be ignored in a production-oriented workflow.

Typical warning themes include:

- unresolved requested packages
- partial runtime policy
- missing module modeling
- unexpected authority source
- legacy-compat fallback usage

A clean run is better than a merely successful run.

---

## How to diagnose common situations

### Case 1 — Wrong package prefix

Look first at:

- detection section
- specs section

Likely causes:

- wrong module environment
- wrong wrapper found first in `PATH`
- fallback module loaded unexpectedly

### Case 2 — Wrong `operating_system` or `target`

Look at:

- facts section
- policy section
- authority section

Ask:

- was the wrong value detected?
- or was detection correct and policy was overridden?

### Case 3 — Too many packages externalized in site `packages.yaml`

Look at:

- policy section
- configuration file

Then confirm:

- `site.external_promotion_mode`

Use:

- `all` for broad promotion
- `providers-only` for conservative promotion

### Case 4 — Template file looks wrong

Look at:

- template section in config
- policy section
- final `configs/templates/<template>/spack.yaml`

Confirm:

- template name
- specs list
- compiler projection

### Case 5 — Output is technically generated but semantically wrong

This is the most important case.

Read the report in full and compare:

- what the machine really is
- what the report says the facts are
- what the report says the policy is
- what the final YAML files contain

Most real issues come from confusion between these three layers.

---

## Practical workflow when something looks wrong

Use this order:

1. inspect `detection-report.txt`
2. inspect `configs/sites/<site>/compilers.yaml`
3. inspect `configs/sites/<site>/packages.yaml`
4. inspect `configs/sites/<site>/config.yaml`
5. change config or overrides only after understanding the report

Do not start by editing generated YAML files manually.
That hides the real issue instead of fixing it.

---

## What a good report should tell you

A good report should allow you to answer these questions clearly:

- What did the tool detect?
- What did it validate?
- Which assumptions did it make?
- What final policy did it derive?
- Which values came from detection?
- Which values came from configuration?
- Which values were overridden?
- Are there warnings that make the outputs risky to use?

If the report does not answer those questions, the problem is not only in the machine. It is also in the observability of the tool.

---

## Summary

`detection-report.txt` is not just a log.
It is the diagnostic explanation of the bootstrap result.

For real operational use, the right mindset is:

- trust the report first
- understand the policy second
- inspect the YAML outputs third

That sequence keeps the debugging process honest and reproducible.
